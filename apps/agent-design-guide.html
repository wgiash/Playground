Manual

The system, as rules.

An operational reference for the Aither wireframe system — the studio's scaffold for producing high-fidelity, brandable wireframes that can become finals. Every token, scale, atom, and pattern has exactly one canonical use. This document is where that use is recorded.

The audience is two-headed. Humans read this to review the system. Agents read this to compose inside it without re-deriving conventions every prompt. Both should be able to find a single answer to "what does --hair resolve to in dark mode" or "which heading do I use here" without parsing source files.

Source of truth. If something on this page disagrees with system/tokens.css, tokens.css wins and this page is wrong — regenerate it. If something here disagrees with CLAUDE.md, CLAUDE.md wins. This document is downstream of both.

How this manual is organised

  • Part I — Orientation. Quickstart, what Aither is, the standing rules, and the atomic taxonomy.
  • Part II — Foundations. Color, typography, spacing, grid, radii, motion, modes, and the brand layer.
  • Part III — Components. Atoms, their states, action hierarchy, and iconography.
  • Part IV — Composition. Frames, recipes, pattern templates, data display, and flows.
  • Part V — Language. Voice and microcopy.
  • Part VI — Appendix. Naming conventions, file map, shipping checklist, and anti-patterns.

How to read

  • Rule blocks (left-bar pull-outs) are non-negotiable.
  • Do / Never pairs show the legal form next to the most common wrong form.
  • Mono labels (Space Mono) name tokens, files, and class selectors. If it's mono, it's a literal that exists in the codebase.
  • Cross-references in body copy use named sections (e.g. §Color, §Atoms), so they survive renumbering.

Quickstart.

To compose a wireframe, do these things in this order. Each step links to the chapter that covers it. If you can't complete a step, stop and read that chapter before continuing.

01
Pick the pattern. Match your surface to dense list, editorial detail, or multi-step form. If it doesn't fit one, surface the gap before inventing. → §Pattern templates
02
Open the four frames. Every surface ships as four frames: web 1440 × 900 and mobile 390 × 844, each in dark and light. Wrap each in .wf[data-mode]. → §Frames & pairing · §Modes
03
Use only tokens. Color via the wireframe-mode tokens (--bg, --fg, --panel, --input, --hair) inherited from the .wf wrapper. Type via the role ladder (Display → Caption). Spacing via --s-*. Radii via --r-*. Never raw hex, raw px, or a font name. → §Color · §Typography · §Spacing · §Radii
04
Compose with atoms. .wf-* primitives are the only legal building blocks at the wireframe stage. Frame chrome is .ki-*, frozen. → §Atoms · §Iconography · §Naming conventions
05
Pick one primary action. Each surface has exactly one data-primary button. Secondaries are outlined; tertiaries are inline text. → §Action hierarchy
06
Run the shipping checklist. Paired? Token-bound? Atomic? Mode-clean? One primary? Auth + error (flows)? Anti-patterns scanned? If anything's no, the frame isn't done. → §Shipping checklist · §Anti-patterns

Brand color and photography come later. The wireframe stage uses bone, dark, and hairlines only. Per-client branding is a separate layer applied to the approved wireframe — see §Brand layer.

Manual

About Aither.

Aither is a design and technology studio. We work directly with founders, emerging brands, and large companies across every space — consumer, enterprise, government, aerospace. Past partners include Google, Sony, Spotify, the U.S. Treasury, and USAID. Past projects include Partiful, Singuli, Flow Labs, Sentropy, Goldcast, and Descartes Labs.

This library exists so we can produce high-fidelity, Aither-branded wireframes that can quickly become finals. The system is the through-line between a brief and a shipped product. The output is paired (web + mobile, dark + light), grounded in tokens, and faithful enough to brand that the jump from wireframe to launch is incremental, not a redesign.

Frame of mind

Compose for a real product, not a placeholder.

Why. These wireframes are meant to ship. Generic stand-in content ("Lorem ipsum", "Item 1, 2, 3") makes the wireframe untestable and re-anchors the visual hierarchy when real copy arrives.

How. Use plausible domain content for whichever product you're composing for. The example wireframes in templates/ use event-domain placeholder content because that was the test-batch subject — it is not a real client surface. A finance dashboard, a CRM, a marketplace, a control panel all live equally inside this system as long as they speak in the brand posture below.

Brand posture

  • Future-making, not loud. Aither's positioning is "design partners enthused in future-making, effusive in the process of co-creating." That tone shows up as: confidence, restraint, no novelty for novelty's sake.
  • Hierarchy by weight and size, never color (Rule 04). The page should feel like a printed editorial, not a marketing site.
  • Bone over white. Light mode is bone (#EAE4E2), never pure white. Pure white reads as a different brand.
  • Minimal chrome. Carry only the chrome the surface actually needs. No ornamentation, no decorative dividers, no breadcrumbs unless the IA earns them.
  • Editorial density. When in doubt about list density, treat the row like a tracklist (YT Music) or a magazine table of contents — meta on a second line, leading mark, breathing room between groups.

Anti-references

What an Aither wireframe should not look like: a generic SaaS template, a Material Design dashboard, a fintech screen with color-coded statuses, a social app with brand-color CTAs and gradient pills. If a referenced pattern would put the work there, surface the conflict before composing.

Studio shorthand

Where useful, lift voice from aither.co: "shipping products, brands, and transformations for legacy-makers" · "future-spired" · "ambitious founders and teams." Use sparingly and only when authentic — don't paste studio copy into a product.

Manual

Standing rules.

Seven rules that override everything else. They are the operational form of CLAUDE.md — same numbering, same order. If a request would violate one of these, surface the conflict instead of silently breaking the system.

Rule 01

Tokens only. Never introduce a new color, font, or styling token.

Why. Token sprawl is the fastest way to break consistency in a system this small. Every accidental hex code is a future inconsistency.

How. If a value needs to live somewhere, pull it back to the closest existing token in system/tokens.css. If no token fits, raise the gap rather than inventing one.

Rule 02

Don't touch infrastructure.

Why. The pan/zoom logic, canvas chrome, comment system, shared menu component, and base slide CSS are tuned. Edits ripple silently.

How. Never modify system/canvas.css, system/chrome.css, the pan/zoom in system/app.jsx beyond registering new sections, the comment system in shared/comments.js, the shared menu component (shared/ds-menu.css / shared/ds-menu.js), or the base .slide / .deck CSS in PRD templates. Add new content under scoped classes; never override the frame.

Rule 03

Respect the atomic taxonomy.

Why. The canvas is organized Foundations → Atoms → Molecules → Organisms → Patterns. Forcing a new item into the wrong layer rots the whole structure.

How. When adding to the canvas, decide which layer the item belongs to (see §Atomic taxonomy), and place it under the correct labelled header. If it doesn't fit a layer, flag it instead of forcing it.

Rule 04 · studio convention

At the wireframe stage, no color. Emphasis by weight and size only.

Why. Wireframes communicate hierarchy through type and spacing — not hue. Color introduced this early reads as branding intent, which short-circuits the review of structure. The brand layer (§Brand layer) is where color enters, after the wireframe is approved.

How. Compose with bone and dark via the wireframe-mode tokens (--bg, --fg, --panel, --input, --hair) inherited from the .wf wrapper. Warm and cool tones in §Color exist as system colors but are not used to indicate state, severity, or selection at this stage. Selection = filled background; severity = position; state = label.

Rule 05

Flows always include auth and error states.

Why. Happy-path flows hide every interesting design decision.

How. Any flow added to the Flows zone must show its auth path(s) and at least one error/failure state in addition to the happy path. If a source template is missing them, surface the gap rather than shipping the flow without them.

Rule 06

No OS or device chrome in wireframes.

Why. Device chrome belongs to the platform, not the product. Drawing it ages the wireframe to a specific moment (today's iOS), distracts from the surface under review, and quietly biases the design toward platform conventions instead of product clarity.

How. No iOS / Android status bar (carrier · time · battery), no notch or Dynamic Island, no home indicator, no Mac menu bar / Touch Bar, no browser address bar, no scrollbars drawn into the frame. The mobile frame starts at the app's first pixel; the device is implied by the 390 × 844 ratio, not rendered.

Rule 07

Keep canvas maintenance current.

Why. The wireframe canvas is read at zoom — labelled header rows are the navigation. The IntroCard is the entry point for every new agent and reviewer. Both rot the moment they fall behind the actual structure.

How. Preserve zoomed-out readability of labelled header rows when adding to the canvas. If a change alters the taxonomy, canonical file, or growth rules, update the IntroCard and DsIntro in apps/wireframe-library.html — mirrored in system/app.jsx and system/app.combined.jsx.

Agent architecture.

How the studio actually produces work. A small crew of agents — one product manager, a flexible number of builders, one auditor — coordinates around a single orchestrator. Canvas, scope, and crew size are decided in the first conversation with the PM.

The first conversation with the PM

Before any building starts, the user talks to the product manager to settle three things:

  • Output location. Where does the work land — Paper, the Playground canvas, a specific repo file, somewhere else? The PM picks the canvas with the user; builders never assume.
  • How many builders. One for a tight, single-surface brief. Two for parallel split work. Three or more for larger briefs where the seam can be drawn cleanly.
  • How the builders relate. Either all on the same thing (each composing a variant of the same surface, so the user can compare options) or split across different things (each owning a non-overlapping slice). The PM decides the seam with the user, never alone.

The crew

  • Product manager. The orchestrator. Holds the first conversation with the user, decides canvas / crew size / seam, spins up the builders, runs the auditor when work is in, and relays notes back. Does not draw.
  • Builders. One or more product designer agents, created by the product manager for the brief. Working in parallel, either on the same thing (variants) or on different things (split scope), depending on what the PM agreed with the user.
  • Auditor. One agent whose only job is to double-check completed work against the system (tokens, taxonomy, frame pairing, voice) and the user's actual needs. Does not edit; writes notes and hands them to the PM.

The loop

01
PM creates the builders. After the first conversation with the user, the product manager spins up the agreed number of builders as high-level product designer agents, with their scopes (variant or split) and the relevant chapters of this manual.
02
Builders compose in parallel. Each builder ships into the canvas the PM agreed with the user, and signals the PM when their slice or variant is complete.
03
PM runs the auditor. Once the builders both report complete, the PM runs the audit agent across the full deliverable — not per-builder, so cross-builder consistency gets caught.
04
PM relays notes. Audit notes go back through the PM to the builders. Builders never receive notes directly from the auditor — the PM mediates so scope and priority stay coherent.
05
Iterate, then ship. Loop steps 02–04 until the auditor signs off. The PM is the only agent that hands the work back to the human.
Rule

The product manager is the only agent with a write channel to the human, and the only agent that talks to the auditor.

Why. Without a single mediator, builders interpret audit notes inconsistently and the human gets four parallel narratives. One mouth to speak through keeps the loop legible.

How. Builders surface blockers up to the PM. The auditor reports to the PM. The PM is the only agent that posts the deliverable, summarises progress, or asks the human a clarifying question.

Atomic taxonomy.

The canvas grows along five layers, left-to-right and top-to-bottom: Foundations → Atoms → Molecules → Organisms → Patterns → Flows. Every new item goes into exactly one of them, under the labelled header for that layer.

Foundations
The non-rendered system layer: color, type, spacing, radii, motion, grid. Documented here in §Color–§Modes. New tokens are forbidden (Rule 01).
color · type · spacing · radii · motion · grid · modes
Atoms
Single-purpose primitives that take one prop and render one thing. Live in system/tokens.css as .wf-*.
.wf-kicker · .wf-meta · .wf-label · .wf-pill · .wf-btn · .wf-input · .wf-hair · .wf-dot · .wf-placeholder
Molecules
Atoms composed into a labelled unit. They have a name, an anatomy, and one or two states.
labelled input · pill row · row item · empty state · search bar · meta line
Organisms
Self-contained surfaces, several molecules deep. They occupy a frame and have their own header/scroll/footer.
nav · table · modal · onboarding step · settings panel · dense list
Patterns
Page-template patterns — reusable composition shells that define a whole surface. In Atomic Design these are sometimes called templates; we keep the name Patterns for continuity with the canvas. Currently three: dense-list, editorial-detail, multistep-form. Source under templates/.
dense-list · editorial-detail · multistep-form
Flows
End-to-end user journeys that thread through patterns and surfaces. Each flow must include auth + error states (Rule 05). Stack vertically on the right of the canvas.
first-time user · return user · recovery · sign-out

Typography.

TWK Lausanne Pan is the only face inside wireframes, licensed locally at 300, 550, and 700. Space Mono labels the system. Caveat is for canvas annotations only. No Apercu, no Inter — Apercu is display-only on the home tile, Inter is not in the system at all.

Three weights, one face

Every weight token in this system resolves to Lausanne 300, 550, or 700.

300 — Body. Default body, lede, meta, captions. --w-body, --w-meta.

550 — Medium. UI, labels, H2, H3. --w-medium, --w-ui, --w-title.

700 — Bold. Display, H1, strong, primary actions when a heavier hit is needed. --w-bold.

How to apply. Use the weight token (var(--w-…)), not the integer. The token may be re-tuned later; integers can't.

A 4px base scale. Use the named tokens — never a raw px value in production styles. The scale is intentionally sparse: one token per role, not one per pixel. If you find yourself reaching for 10px, use --s-2 or --s-3.

Composition rules

  • Inside an atom (button, pill, input): --s-2 to --s-4 for padding; --s-2 for icon-text gap.
  • Between atoms in a molecule (label + input, kicker + heading): --s-2 to --s-3.
  • Between molecules in an organism (form rows, list rows): --s-4 to --s-6.
  • Between organism sections on a page: --s-12 to --s-16.
  • Between top-level sections (between H1 sections of a doc): --s-20 to --s-24.
  • Frame side margin on the canvas: 24px (locked, see §Grid).
Do padding: var(--s-4) var(--s-6); gap: var(--s-3); margin-block: var(--s-12);

Token-bound. Reads at every zoom; survives a token re-tune.

Never padding: 17px 25px; gap: 11px; margin: 50px 0;

Off-scale values. Sprawl. Impossible to maintain.

Two grids, one per surface family. The numbers below are locked from the canvas IntroCard (system/app.jsx) and apply to every Aither wireframe in this library.

Mobile

Common spans

  • Hero / full-bleed surface — 12 of 12 (web) · 4 of 4 (mobile).
  • Editorial body / form — 8 of 12 centred (web) · 4 of 4 (mobile).
  • List + detail (paired) — 5 + 7 (web). Mobile collapses to 4 of 4 stacked.
  • Sidebar layout — 3 + 9 (web). Sidebar fixed, content scrolls.
  • KPI row — 3-col or 4-col equal-width (web). Mobile stacks 1-up.

Four radii. Two are structural (sharp + slightly rounded). One is for inputs. The pill is reserved.

Foundations

Motion.

One duration, one easing curve. Everything that moves uses these. The curve favors a fast start and a long tail — feels like a heavy object settling, not a tween.

--dur   280 ms    --ease   cubic-bezier(0.32, 0.72, 0, 1)

When to animate

  • Hover/focus on interactive atoms — color, border-color, background only.
  • Mode change (dark/light) — fade panel + foreground, never animate type sizes.
  • Sheet / modal entry — slide + fade together with one --dur.

When not to

  • Layout transitions inside a list (rows reordering, items adding) — feels noisy, doesn't earn its frame budget.
  • Zoom-in animations on the pan/zoom canvas — the user controls the camera; we never do.
  • Decorative loops (pulses, breathing, rotating). The system does not pulse.
Do transition: color var(--dur) var(--ease), border-color var(--dur) var(--ease);

Both tokens. One duration, one curve.

Never transition: all 0.2s ease-out;

Bypasses tokens; all animates layout properties you didn't intend.

Foundations

Modes.

Two modes, paired everywhere. Light is the default — what users see. Dark is the companion mode, with a full set of token values defined for every atom. Every wireframe ships in both, and the canvas displays them side-by-side.

The mode lives on the outer .wf wrapper as data-mode="dark" or data-mode="light". Atoms inside the wrapper inherit --bg, --fg, --fg-soft, --fg-mute, --panel, --input, --hair, --hair-2, and --placeholder from it. Atoms never hardcode a palette.

<div class="wf" data-mode="dark">
  <span class="wf-kicker">Section</span>
  <button class="wf-btn" data-primary="true">Continue</button>
</div>

<div class="wf" data-mode="light">
  <span class="wf-kicker">Section</span>
  <button class="wf-btn" data-primary="true">Continue</button>
</div>

Both blocks render the same atoms; the wrapper supplies the palette. To restyle, retheme, or pair a frame in both modes, change the wrapper, never the atom.

Do .wf-pill { border-color: var(--hair-2); color: var(--fg); }

Atom inherits palette via mode tokens. One CSS rule, two themes.

Never .wf-pill { border-color: rgba(234,228,226,0.22); } .wf-pill[data-mode="light"] { border-color: rgba(18,13,11,0.28); }

Mode hardcoded into the atom. Doubles every rule; breaks pairing.

Foundations

Brand layer.

A wireframe in this library is the bone of a project. The brand layer is the meat — accent color, photography, brand typeface, micro-illustration, motion accent — that lives on top of the wireframe to make it that specific client's product. The two are sequential, not contradictory.

Sequence

  1. Compose the wireframe in this system. Bone, dark, hairlines only. Approve the structure.
  2. Branch to a project surface (a separate file under apps/, or a project-specific repo).
  3. Layer the brand on top. Add a project-specific tokens file that extends system/tokens.css — never replaces it.
  4. Cross-check the branded surface against this manual. Atom anatomy, action hierarchy, naming, type ladder all stay system. Color + display typography + photography become project.

What stays system, what becomes project

System
Type role ladder · spacing scale · radii · motion curve · atomic anatomy (.wf-* primitives) · action hierarchy · naming conventions · file map · grid.
Never overridden by a project
Project
Accent color(s) · photography · illustration · brand display typeface (headlines only — body always falls back to --font-ui) · domain copy · custom motion accents.
Lives in a project tokens file, never in system/tokens.css
Rule (brand layer)

A project's brand layer extends the system; it never replaces it.

Why. Replacing the system means we lose the through-line between the wireframe and every other Aither project. Extension keeps the system shape and the brand voice both intact.

How. Add a project tokens file. Define only the new tokens (--accent, --brand-display, etc.) and the few overrides the brand needs. Reference them from project-level CSS. Re-import system/tokens.css first so all base tokens still resolve.

Brand display typeface

If a project's brand has a display typeface (e.g. a custom Apercu, a Pangram, a project-specific cut), it lives at the Display and H1 roles only. H2 onwards stays on --font-ui. This keeps body legibility consistent across every project the studio ships.

Rules

  • Default always renders. Inherits palette from parent .wf[data-mode].
  • Hover brightens the border to --fg. Web only.
  • Focus is a 2px outline at --fg with 2px offset. Always visible — never outline: none. Keyboard users depend on it.
  • Active / On = filled background with foreground inverted. Use data-active="true" on toggleables and data-primary="true" on the primary action.
  • Disabled = the same atom + disabled attribute + aria-disabled="true" + cursor: not-allowed + opacity 0.4. All four — opacity alone isn't enough to communicate the state to assistive tech.

Additional input states

Inputs and buttons carry two more states beyond the matrix above. Both are valid wireframe-stage states (the form has to render its own loading and error frames).

Loading Async work pending. The atom keeps its outline; foreground dims; the action label is replaced with a verb in progress ("Continuing…"). No spinner inside the wireframe — copy carries the state.
Error Validation failed. Border on the input switches to --fg at full strength; a 12px error message in mono sits below it. No red. Severity carried by position and copy.

Action hierarchy.

Every surface has exactly one primary action. This is not a guideline — it's the load-bearing rule for surface clarity. If two candidates feel primary, one of them isn't.

Tertiary

Destructive
Delete account Tertiary form · the copy earns the weight · never red

Rules

  • One primary per surface. Modals, sheets, and inline forms each count as their own surface.
  • Secondary is the alternative path that's still expected — "Save draft", "Cancel", "Back".
  • Tertiary is text-only and lives inline within prose or as a row action. Never use a tertiary as the only action on a surface.
  • Destructive uses the tertiary form at the wireframe stage. The copy carries the weight ("Delete account", "Cancel subscription"). Don't introduce red here. The wireframe never does.)
  • CTA placement. Web: bottom-right of the form / top-right of the surface. Mobile: pinned to the bottom edge inside a 16px-padded sticky bar.

Iconography.

Lucide-style line icons. 1.5px stroke, 24×24 viewBox, default render at 18px. Stroke uses currentColor so icons inherit from their parent — never set an icon's color directly.

The full set lives in system/icons.jsx as window.Icons. Use what's there before drawing a new one. If you need an icon that isn't in the set, raise it for review rather than inlining a one-off SVG.

Set

navigation : Back · ChevL · ChevR · ChevD
search     : Search · Filter
actions    : Plus · Check · More · MoreH · Download · Share · Heart · Bookmark · Pin · Pin2 · Play · PlayOutline
content    : Calendar · Map · Globe · User · Info
status bar : Signal · Wifi · Battery   (filled, mobile only)

Rules

  • Size. Default 18px. Tap targets at 20–24px. Status bar icons at 14px. Never below 12px.
  • Stroke. 1.5px on all line icons. Status-bar icons (Signal/Wifi/Battery) are filled, not stroked.
  • Color. Always currentColor. To dim an icon, dim the parent. Never fill="#…".
  • Alignment. Pair an icon with text using display: inline-flex; gap: var(--s-2). Vertically centre on the cap height of adjacent text, not the line height.
  • Decoration. Don't decorate (no badges, no gradients, no drop shadows). The line is the icon.

Never <svg width="20" stroke="#EAE4E2" stroke-width="2"> … </svg>

Off-spec stroke, hardcoded color, one-off SVG.

Frames & pairing.

A frame is a single rendered surface — one screen, one mode, one device. Every frame on the canvas is wrapped in .wf[data-mode] (§Modes), positioned by the canvas, and labelled with a kicker + title above it.

Canonical sizes

Web
1440 × 900. Desktop viewport. 12-col grid, 24px gutter, 24px side margin (§Grid).
1440 × 900 · 12-col · 24/24
Mobile
390 × 844 (iPhone class). 4-col grid, 24px gutter, 24px side margin. Frame starts at the app's first pixel — no iOS / Android status bar (Rule 06).
390 × 844 · 4-col · 24/24

The pairing rule

Studio convention · pairing

Every surface ships in the agreed mode set, paired across web and mobile.

Why. Light is the production default and the studio's default deliverable. Dark is fully tokenized — every atom resolves in both modes — but it's only produced when the human asks for it. Web and mobile are always paired so we never design one in isolation.

How. The PM agrees the mode set with the human in the first conversation. Three valid sets:

  • Light only (default): web-light + mobile-light = 2 frames per surface.
  • Dark only (only when asked): web-dark + mobile-dark = 2 frames per surface.
  • Both (only when asked): web-light + web-dark + mobile-light + mobile-dark = 4 frames per surface.

Within the agreed set, missing a frame means the surface isn't done. Extra frames in modes the human didn't ask for are also wrong — they waste builder time and clutter the deliverable.

Density

  • Lists. Row height 56–72px on web, 64–76px on mobile. Editorial rhythm — meta on a second line, leading mark at 32–40px, trailing action at 28–32px.
  • Forms. Field height ~44px (input padding 12/14). 16–24px vertical between fields.
  • Cards. 24–32px internal padding. Cards never sit alone — they're either gridded or stacked with a hairline.

Flows.

A flow is a labelled sequence of frames stacked vertically on the right of the canvas. Every flow represents one user journey from a starting state to a steady state. Every flow ships with auth and error states (Rule 05).

What every flow must contain

  • Entry frame — where the user starts (signed-out, signed-in cold, or returning).
  • Auth path(s) — sign-in, sign-up, or session-recovery, whichever the journey actually crosses.
  • Happy path — the steady-state sequence.
  • At least one error / failure state — invalid input, network failure, expired session, denied permission. Pick the one most likely on this journey.
  • Exit / steady state — what the screen looks like after success.
Rule 05 (restated)

If a flow's source template is missing auth or error states, surface the gap rather than shipping the flow without them.

The right move is to write the missing frames yourself if you have what you need, or to flag them as TODO with a one-line note about what they should contain.

Frame labelling

Every frame in a flow has a label above it: NN — Surface · State. Numbers are sequential within the flow. Surface is a noun (Sign in, Dashboard, Settings). State is a state (cold, with-data, error, success).

01 — Sign in · cold
02 — Sign in · validating
03 — Sign in · error · invalid email
04 — Dashboard · cold
05 — Dashboard · with-data
Manual

Voice & microcopy.

The system speaks in short, declarative sentences. Headlines end in periods. Kickers are categorical, not editorial. Errors are dignified, not apologetic.

Kickers

  • Format: Group · Number or Group · Subgroup. Period-separated, never commas.
  • Examples: Foundations · 04 · Patterns · 11 · Sign in · cold · Updated · 2026-04-24
  • Never use a kicker as a sentence. Never punctuate with a period at the end.

Headlines

  • Always end with a period. "Color." not "Color".
  • Keep to ~20 characters. Two short sentences beat one long one.
  • Lowercase nouns ("color", "spacing") when the kicker has already named the category.

Body

  • Lead with the rule, then the reason.
  • ~64ch line length. Two short paragraphs beat one long one.
  • Forbidden words in product copy: just, simply, oops, woops, uh-oh, exclamation marks.

Empty & error states

  • State the situation in five words. "Nothing to show yet." · "Couldn't reach the server."
  • One next-step sentence. ~32ch.
  • One action button. Secondary by default — primary only if there's a single right move.
  • Never apologetic, never cute.
Do Couldn't reach the server. Try again in a moment. [ Retry ]

Five-word headline. One next step. One action.

Never Oops! Something went wrong 😞 We're so sorry, please try again or contact support@… [ Try Again! ] [ Contact Support ]

Apology, emoji, exclamation, two CTAs. Off-voice on every count.

Manual

Shipping checklist.

A frame is not done until every item below is true. Run through this list before declaring a surface shipped, and run it again whenever a frame is touched. Done is a state, not a verb.

Paired. The surface ships as four frames: web-dark · web-light · mobile-dark · mobile-light. Missing a quadrant means the surface isn't done. (§Frames & pairing)
Token-bound. No raw hex, no raw px, no font names — every value goes through var(--token). Search the file for #, px outside tokens, and font literals. (§Color · §Typography · §Spacing)
Atomic. Built from .wf-* atoms and .ki-* chrome only. No bespoke component CSS introduced for this frame. (§Atoms · §Naming conventions)
Mode-clean. Atoms inherit palette from the .wf[data-mode] wrapper. No hardcoded fg / bg / border colours inside an atom. (§Modes)
One primary. Each surface has exactly one data-primary action. Two means one of them isn't primary. (§Action hierarchy)
States covered. The frame renders the right state for its moment. If it's an interactive surface, default + hover + focus are accounted for in the prototype. (§States)
Auth + error. If this is part of a flow, the flow includes auth path(s) and at least one error / failure state. (§Flows · Rule 05)
Labelled. The frame has a kicker + title + subtitle above it on the canvas (NN — Surface · State).
Voice on. Headlines end in periods. No just, simply, oops, exclamation marks. Domain content is plausible, not placeholder. (§Voice & microcopy · §About Aither)
Brand-layer separated. No project-specific color, photography, or display typeface inside the wireframe. Those live in the brand layer once the wireframe is approved. (§Brand layer)
Anti-patterns scanned. Run through §Anti-patterns. If any are present, fix or flag.
Responsive at every breakpoint. Web frame works at 1440 / 1280 / 1024. Mobile frame works at 390 / 360. No content clipped, no chrome overlapping content.
Manual

Anti-patterns.

A consolidated list of moves that are wrong inside this system. If you find yourself reaching for one of these, stop and reach for the recipe instead.

Never

Hardcoded hex codes in component styles.

Always var(--token). The hex below is the same color but unsubscribed from the system.

Never

Apercu Pro or Inter inside a wireframe.

Apercu is display-only and lives on the home tile. Inter is not in the system at all. Wireframes use --font-ui (Lausanne 300 / 550 / 700) and --font-mono (Space Mono) — nothing else.

Never

Color used to indicate state, severity, or selection in a wireframe.

Use weight, size, position, or filled-fg inversion. Color in a wireframe reads as branding intent (Rule 04).

Never

A new H1/H2 size, a new spacing value, a new radius.

The scale is sparse on purpose. Pick the closest existing token (§Typography, §Spacing, §Radii).

Never

Modify system/canvas.css, system/chrome.css, shared/comments.js, or pan/zoom logic in system/app.jsx.

Frozen infrastructure (Rule 02). Add new content under scoped classes; don't edit the frame.

Never

Hand-edit <nav id="ds-menu-nav"> inside a deck, or write slide numbers into .tag / .page.

The shared menu rebuilds the nav and the deck runtime stamps the numbers on load — your edits are discarded.

Never

Animate all, animate layout properties, or invent a duration.

One --dur, one --ease, on color/border/background/transform only (§Motion).

Never

Ship a flow without auth and at least one error state.

Rule 05. If they're missing from the source, write them or flag the gap.

Never

Apologetic, cute, or exclamatory copy.

Wireframes communicate in short declarative sentences. No "oops", no exclamation marks.

Never

Force an item into the wrong taxonomy layer.

If a thing doesn't fit Foundations / Atoms / Molecules / Organisms / Patterns / Flows, raise it for review (§Atomic taxonomy).

Never

Render OS or device chrome inside a wireframe.

No iOS / Android status bar (carrier · time · battery), no notch or Dynamic Island, no home indicator, no Mac menu bar / Touch Bar, no browser address bar, no scroll bars drawn into the frame. Wireframes show only the product surface. The mobile frame starts at the app's first pixel — the surrounding device is implied by the 390 × 844 ratio, not drawn.
