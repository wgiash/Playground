Product · 1 · Framing

Sunpull · Product Spec.

v1.0 · 2026-04-24. DRI Jasmine; built by Michael. Prereq Master Sunspell PRD signed (in flight — portfolio pass pending). Gates unlocked on full signoff: Brand Spec + Tech Spec + Roadmap Spec may begin.

Product · 2 · Framing

North star.

Two framings, both true. Sunpull is a B2B memory product — subscribable standalone today, context substrate for the unified Sun OS tomorrow. The consumer sibling (Sumani) ships on the shared backend under a separate brand; Sunpull stays B2B permanently.

  1. Memory artifact for the team's AI. Every Sun and any agent reads from Sunpull so output reflects current team taste, not training-time priors.
  2. Taste profile the team owns. Sunpull makes a team's taste legible, queryable, and portable — instead of scattered across a dozen platforms that own the signal.
Product · 3 · Framing

Thesis.

A taste-driven team's edge is pattern recognition — noticing what's good before it's obvious. Today that signal leaks across LinkedIn saves, Instagram DMs, Pinterest, Are.na, Slack links, iMessage. Sunpull consolidates it. A provider product whose value compounds as more Suns plug in, but meaningful on day one as a rediscovery surface.

Product · 4 · Framing

Problem.

Three specific pains. No solutioning.

  • Founder context can't be reused. Sunrun Runs generate from generic priors instead of taste-encoded refs. Team members can't find what the founder found. The same ref gets rediscovered.
  • AI stays stale. Every LLM call starts from training-time knowledge; nothing connects the agent to what Aether is actually paying attention to this month.
  • The user doesn't own their taste. Instagram, TikTok, Pinterest hold the saves — the user is a tenant of their own preferences.
Product · 5 · Framing

Mental model.

Pinterest masonry grid meets source-native preview — a visual, browseable library where LinkedIn posts render like LinkedIn, Instagram looks like Instagram, web articles render cleanly. Serendipitous rediscovery matters as much as query. Query API is the hidden second surface — consumers (Sunrun, Sundial) read through it; the founder never sees it.

Product · 6 · Framing

Personas.

Primary: high-agency taste-maker on a taste-driven team. Plus role-based access across Aither + two system consumers (Sunrun, Sundial). Key separation: personal vs. company saves, enforced at the query layer.

Product · 7 · Framing

Principles.

Ten non-negotiables from §3. P1–P7 shape UX; P8–P10 shape provider + portfolio discipline. P8/P9 explicitly cite Sunrun/Sundial — candidates to promote to Master Operating Principles.

  1. P1 Capture is frictionless. <10s; zero typing required for basic capture.
  2. P2 Organization is dialogue-based, not forced upfront. Sunpull asks in batches, learns patterns, applies retroactively.
  3. P3 Tags tie to Sunrun workflows. Taxonomy designed with Sunrun in mind; mapping is explicit.
  4. P4 Masonry visual browse. Not a list. Serendipitous rediscovery matters.
  5. P5 Source-native preview. LinkedIn looks like LinkedIn; IG visual intact; web rendered cleanly.
  6. P6 Personal and company separated. Enforcement at the query layer, not just UI.
  7. P7 Progressive disclosure. Simple surface for browse/save; advanced on deeper engagement.
  8. P8 Provider-first discipline. Stable query contract; breaking changes versioned with overlap window.
  9. P9 Scope tight. New ingestion / tags / endpoints require real pain signal. Matches Sunrun P8 / Sundial P8.
  10. P10 The user's saves are Aither's memory. Every choice serves the memory-to-AI pipeline.

Product · 9 · Strategy

Positioning.

Sunpull is the memory layer of Sun OS — a B2B memory product for taste-driven teams. It captures everything worth remembering into a single masonry-grid library that's tagged, searchable, and queryable via an API that other Suns read from. Aether is the first customer; design / creative / strategy shops are the wedge. Sunpull stays B2B permanently — a consumer sibling (Sumani) ships separately on the shared backend.

Product · 11 · Strategy

Dependencies.

Sunpull is a provider — stands alone, but gates unlock when consumers connect. Distinct from Interop (which describes runtime contracts, see Tech).

  • Phase 0 prereq: nothing external. Founder's LinkedIn + Instagram exports, a Telegram bot, Postgres + S3.
  • V1 prereq: Claude API access; Chrome + Arc extension dev sign-in; object storage (R2/S3/Backblaze — open).
  • Sunrun consumer: reads /v1/saves, webhook recipient for save.created / save.tag_changed. Wires in once Sunrun ships.
  • Sundial consumer: reads /v1/saves?tags=...&since=.... Contract live in V1; implementation P1.5.
  • Master PRD: Operating Principles (P8/P9/P10) inherit from Master. Master is currently empty scaffold — signoff on Master blocks Sunpull final signoff.

Not explicit in sunpull_requirements.md. Implicit trigger: M5 breach (founder rediscovery < 1/wk by W8) signals the rediscovery surface doesn't work. M6 breach (no Live Run sources from Sunpull by W8) signals the provider thesis doesn't work. Flag for Jazz — formalize thresholds.

Product · 16 · Scope

Risks.

Primary risks: platform data-access, cost creep, provider-SLO, adoption. Seeded from Tech §13 + Product §11 + §10. Not itemized in Jazz's spec — seed list here; flag for Jazz to rank.

  • LinkedIn/IG ToS drift. Exports are the only compliant channel. Format changes (which LinkedIn has made before) break the parser. Mitigation: version-sniff on header; fail loud.
  • Media cost creep. Heavy IG/video saves grow S3 fast. Mitigation: per-item cap, warn at 100 MB, skip at 500 MB.
  • Claude cost. Soft-warn at 80% of estimated budget per import job (§10 NF6).
  • Provider dependency. Sunpull down degrades Sunrun + Sundial. Target 99.5% uptime (§10 NF2) — higher bar than peers.
  • Adoption / compounding failure. If team never queries and no Sunrun Live Run sources from Sunpull, provider thesis breaks. M6 tracks.

Brand · 8 · Voice

Tech · 2 · Architecture

Architecture.

Five components, all on Render. No scheduler, webhook emitter, query API, admin UI, or Claude worker in Phase 0 — those are V1.

LinkedIn export (.zip)  ───┐
Instagram export (.zip) ───┼─── Ingestion API (Node/Express) ──── Content processor (worker) ──── Postgres (Save, ImportJob)
Text (Telegram webhook) ───┘                                            │
                                                                        └──── S3/R2 (media + staging uploads)

Components

  1. Ingestion API — HTTP endpoints: export uploads + Telegram webhook
  2. Content processor — worker; parses exports, fetches URL previews, downloads media
  3. Postgres — Save + ImportJob entities
  4. Object storage — media blobs + .zip staging holding
  5. Telegram bot — registered via BotFather; webhook → ingestion API
Tech · 3 · Architecture

Data model.

Two tables in Phase 0: saves + import_jobs. All others (tags, archetypes, consumer_subscriptions, etc.) are V1 net-new. The Save schema is the load-bearing decision — get it right now; V1 is mechanical.

saves
  id                 uuid PK default gen_random_uuid()
  source_platform    text check (linkedin|instagram|web|text|manual)
  source_url         text
  title, content_text, preview_image_url  text
  media_urls         text[] default '{}'
  author_name, author_handle  text
  posted_at, saved_at, imported_at  timestamptz
  user_context_note  text
  is_personal        boolean default false
  archived_at        timestamptz
  unique (source_platform, source_url) where source_url is not null
  index (saved_at desc), (source_platform), (is_personal) where is_personal = false

import_jobs
  id                 uuid PK
  source_platform    text
  uploaded_file_key  text          (S3 key of staged .zip)
  status             text (pending|processing|complete|failed)
  items_imported, items_duplicates, items_failed  int
  started_at, completed_at  timestamptz
  error_log          text
Tech · 4 · Architecture

Ingestion / storage.

Three ingestion paths in Phase 0 — LinkedIn export, Instagram export, Telegram text. Media in private object storage; staging .zips auto-deleted after 7 days.

Paths

  • LinkedIn · POST /admin/imports/linkedin · multipart .zip → parse Saved_Posts.csv → upsert on source_url. OG fetch for preview (best-effort).
  • Instagram · POST /admin/imports/instagram · multipart .zip → parse saved/saved_posts.json → copy media to object storage → upsert.
  • Telegram · POST /webhooks/telegram · validated via secret header + sender whitelist → parse URL + context + personal suffix → OG fetch → insert save.
  • iMessage (optional, if time) · POST /webhooks/imessage — same payload shape as Telegram.

Storage

  • Postgres (Render managed) — single DB, daily backups.
  • sunpull-media-<env> · permanent, signed URLs only.
  • sunpull-imports-<env> · staging, 7-day lifecycle auto-delete.
  • Object keys: <platform>/<save_id>/<filename>.
Tech · 5 · Integration

External integrations.

Three live dependencies in Phase 0. Claude is not Phase 0 — wired in V1 Week 1 for tag suggestion + dialogue.

  • Telegram Bot API · api.telegram.org/bot<TOKEN>/…. Webhook secret validated via header.
  • URL preview · open-graph-scraper + metascraper fallback. 10s timeout. Fall back to <title>. Never block ingest on preview failure.
  • LinkedIn / Instagram data exports · user-uploaded .zip. No API; scraping is ToS-prohibited. Re-export periodically to capture new saves.
  • Claude API · NOT Phase 0. Opus for dialogue, Sonnet for tag suggestion (V1).
Tech · 6 · Integration

Auth / permissions.

Founder-only access in Phase 0. No multi-user OAuth until V1. References Master Shared Services — which is currently empty scaffold.

  • Admin endpoints (/admin/imports/*) — Authorization: Bearer <token>, stored in Doppler.
  • Telegram webhook — secret token in header (BotFather-configured) + sender user_id whitelist.
  • Deferred to V1: full OAuth, multi-user, team scoping, per-consumer service-to-service bearer tokens with quarterly rotation.
Tech · 7 · Integration

Cross-product interop.

Phase 0 has no cross-product interop — Sunpull stands alone. V1 wires the provider contract defined in Product §5.9 / §8.7. Sundial implementation deferred to P1.5.

V1 — Provides (Sunpull → Sunrun / Sundial)

  • GET /v1/saves?tags=...&include_personal=false&since=<ISO8601>
  • Webhook: save.created, save.tag_changed. Consumer registers URL + optional tag filter.
  • Per-consumer bearer token, scoped to workspace, rotated quarterly.
  • Rate limit: 120 queries/min/consumer. Webhook retries 3x exponential backoff.
  • Versioning: /v1/*. Breaking changes ship as /v2/* with ≥2-week overlap.

V1 — Consumes (Sunpull → Sunrun, optional)

  • GET sunrun/workflows — populates workflow-key dropdown in tag management. Fallback: manual text input with warning.

Privacy & data handling.

Founder-only in Phase 0 — is_personal flag is recorded per save but team-query enforcement activates in V1 with the consumer API.

  • is_personal recorded on every save; no query-layer exposure yet.
  • Media in private object storage — no public URLs.
  • Deletion: manual via DB + object-storage console in Phase 0. Proper cascade + 30-day soft-delete lands with V1.

Tech · 12 · Future-proofing

Migration path to unified OS.

Phase 0 schema is forward-compatible. V1 additions are all net-new tables referencing saves.id — no Save schema changes. Save is the load-bearing decision.

V1 net-new tables

  • tags, save_tags — tag engine
  • archetypes — taste profile
  • consumer_subscriptions, consumer_query_logs — provider API
  • categorization_sessions — dialogue state
  • save_clusters — embedding groups

V1 net-new services

  • Claude worker (tag suggestion + dialogue)
  • Query API (/v1/saves?tags=…)
  • Webhook emitter
  • Web frontend (read-only against DB initially)
  • Browser-extension backend (reuses content processor)

Non-negotiable in Phase 0

To avoid V1 pain: uuid PKs, source_url uniqueness per platform, is_personal on every save, media_urls as array.

Unified-OS lock-in

Consumer contract lives under /v1/*. Breaking changes ship as /v2/* with ≥2-week overlap; consumers notified in advance.

Tech · 13 · Future-proofing

Tech risks.

Five Phase 0 risks; each has a stated mitigation in §13.

  • LinkedIn export format drift. LI has changed the CSV layout before. Mitigation: version-sniff on header row; fail loud with a clear error — never silently accept wrong-shaped data.
  • Instagram export size. Heavy users have multi-GB exports. Mitigation: stream-unzip to disk (not memory); chunked upload API.
  • Telegram webhook drops. Telegram retries but occasionally loses deliveries. Mitigation: idempotent on message.message_id; log every receipt before processing.
  • URL preview blocks. Many sites block bot user agents. Mitigation: accept save with minimal metadata; never block ingest on preview failure.
  • Media cost creep. Heavy video saves grow storage fast. Mitigation: per-item cap — warn at 100 MB, skip at 500 MB, log.
Tech · 14 · Future-proofing

Open tech questions.

Six Phase 0 opens. Most are Michael's call; Q5 needs founder to confirm Telegram user_id before first deploy.

  1. Object storage — R2 vs S3 vs Backblaze. Cost check at ~10GB + growth. Recommend: R2 (no egress fees).
  2. Queue — pg-boss vs BullMQ + Redis. Recommend: pg-boss Phase 0; BullMQ in V1 if workers demand.
  3. URL preview library — open-graph-scraper vs roll own. Recommend: open-graph-scraper + metascraper fallback.
  4. Content-hash dedup — Phase 0 dedupes on (platform, source_url). Same article from different URLs — defer content-hash to V1.
  5. Founder's Telegram user_id — confirm + whitelist before first deploy.
  6. Admin upload auth — bearer-only via Doppler, or add IP allowlist? Recommend: bearer-only at this scale.

Roadmap · 7 · Decision

Cross-product coordination.

From Product §18 + §8.7 / §8.8. Contracts Sunpull shares with Sunrun and Sundial. Sequencing: contracts live in V1; Sundial implementation P1.5.

  • Sunrun query (Product FR-G1, §8.7) — Sunrun calls GET /v1/saves?tags=…&include_personal=false at Run-execute time. Lands when Sunrun ships and Sunpull V1 is live.
